# Layout Pattern Selection

Layout patterns are structural tools. Choose the pattern that best matches the
information job, not the pattern that sounds visually attractive.

## Selection Flow

1. Identify the section job: orient, compare, prove, collect input, navigate,
   explain a process, or expose operational data.
2. Count the number of distinct ideas or actions.
3. Decide whether order matters.
4. Decide whether items must be compared side-by-side.
5. Decide whether the section needs a primary focal item.
6. Select the simplest pattern that preserves the decision.

## Pattern Guidance

### `centeredHero`

Use when the page has one primary message and one dominant action.

Why: centering reduces competing hierarchy. It is weak when the user needs
comparison, evidence density, or a parallel visual object to understand the
offer.

### `splitHero`

Use when orientation needs two different evidence modes, such as message plus
product proof, form, preview, or trust card.

Why: the split lets the first viewport carry both promise and evidence. Do not
use it when the second column is decorative.

### `twoColumn`

Use for paired explanation, comparison, feature plus detail, or content plus
control.

Why: two columns signal relationship. If the columns are unrelated, the layout
creates false comparison.

### `cardGrid`

Use when items are peers in a repeated set and scanning across items matters.

Why: grids make breadth visible. They are weak for ordered processes, primary
decision paths, or content where one item is clearly more important.

### `stack`

Use when order matters or the user must process one idea at a time.

Why: vertical order creates narrative and dependency. It is stronger than a grid
for steps, arguments, setup flows, and progressive disclosure.

### `sidebar`

Use when persistent navigation, filters, or context must stay adjacent to a
changing main region.

Why: a sidebar supports repeated task switching. It is overkill for one-off
supporting copy.

## Structural Rules

- Do not use a grid to hide uncertainty about priority.
- Do not use a split layout unless both sides have jobs.
- Do not use a carousel or tabs when all content must be compared at once.
- Do not encode visual decoration as layout structure.
